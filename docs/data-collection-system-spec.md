# 텍스트 마이닝 시스템 아키텍처 설계

## 시스템 개요

다양한 소스(뉴스, 문서, API, 위키피디아 덤프)에서 텍스트 데이터를 수집하고, 자연어 처리를 통해 단어 빈도를 분석하며, 자동으로 카테고리를 분류하는 시스템

## 핵심 목표

1. **대용량 텍스트 처리** - 위키피디아 덤프 등 GB 단위 파일 처리
2. **실시간 데이터 수집** - 뉴스, API 등 동적 데이터 수집
3. **한국어 자연어 처리** - 형태소 분석, 단어 추출
4. **빈도 기반 인사이트** - TF-IDF, 단어 빈도 분석
5. **자동 카테고리 분류** - 텍스트 기반 자동 분류

## 시스템 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                         Portal (API Layer)                  │
│                    REST API / Query Interface               │
└─────────────────────────────────────────────────────────────┘
                                │
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│   Harvest    │       │    Smithy    │       │    Codex     │
│   (수집)      │──────▶│   (가공)      │──────▶│   (저장/색인) │
└──────────────┘       └──────────────┘       └──────────────┘
│ - 크롤링       │       │ - 텍스트 정제  │       │ - 색인        │
│ - API 수집    │       │ - 형태소 분석  │       │ - 검색        │
│ - 파일 처리   │       │ - 빈도 계산   │       │ - 캐싱        │
│              │       │ - 카테고리 분류│       │              │
└──────────────┘       └──────────────┘       └──────────────┘
```

## 데이터 흐름

```
[원시 데이터] → [수집] → [정제] → [분석] → [저장] → [서비스]
     ↓           ↓        ↓        ↓        ↓         ↓
  뉴스/문서    Harvest   Smithy   Smithy    Codex    Portal
```

## 모듈별 책임

### Harvest (데이터 수집)

- **역할**: 다양한 소스에서 원시 데이터 수집
- **핵심 기능**:
    - 웹 크롤링
    - API 통합
    - 대용량 파일 스트리밍 처리
    - 스케줄링

### Smithy (데이터 가공)

- **역할**: 수집된 데이터를 의미있는 정보로 변환
- **핵심 기능**:
    - 텍스트 정제 (HTML 제거, 정규화)
    - 한국어 형태소 분석
    - 단어 빈도수/TF-IDF 계산
    - 카테고리 자동 분류

### Codex (데이터 저장/색인)

- **역할**: 가공된 데이터 저장 및 빠른 검색 지원
- **핵심 기능**:
    - 데이터 영구 저장
    - 효율적인 색인 구조
    - 빈도순 정렬
    - 캐싱

### Portal (API 서비스)

- **역할**: 외부 인터페이스 제공
- **핵심 기능**:
    - REST API
    - 데이터 조회
    - 실시간 모니터링

## 핵심 데이터 모델

```
Document (원본 문서)
    ↓
Word (단어 사전)
    ↓
WordFrequency (빈도 정보)
    ↓
Category (분류 정보)
```

## 기술 스택 개요

- **언어**: Kotlin + Spring Boot
- **데이터 처리**: Coroutines (비동기), Stream API (대용량)
- **자연어 처리**: Open Korean Text
- **저장소**: JPA + H2/PostgreSQL
- **메시징**: Kafka (대용량 처리시)

## 확장성 고려사항

1. **수평 확장**: 각 모듈 독립 배포 가능
2. **메시지 큐**: 모듈 간 비동기 통신
3. **캐싱 전략**: 빈번한 조회 데이터 캐싱
4. **배치 처리**: 대용량 파일은 청크 단위 처리

## 주요 성능 목표

- 시간당 10,000+ 문서 처리
- API 응답 시간 < 200ms
- 메모리 효율적 대용량 파일 처리
- 95%+ 형태소 분석 정확도

## 구현 전략

### Harvest 모듈 구현 방식
- **파일 처리**: Spring Batch를 사용한 대용량 파일 스트리밍
- **라인 단위 읽기**: FlatFileItemReader로 메모리 효율적 처리
- **데이터 전송**: 카프카 프로듀서를 통해 원시 라인 데이터 발행
- **XML 파싱 제외**: 구조적 파싱 없이 라인 단위 전송으로 처리 분산

### 처리 분산 원칙
```
Harvest: 파일 I/O + 카프카 발행 (단순, 빠른 처리)
   ↓
Smithy: XML 파싱 + 텍스트 정제 + 형태소 분석 (복잡한 NLP 로직)
   ↓
Codex: 저장 + 색인 + 검색 최적화
```

### 키워드 추출 중심 설계
- **목적**: 단어 빈도 분석을 위한 키워드 추출
- **XML 처리**: 완전한 구조 파싱 불필요, `<text>` 태그 내용만 추출
- **효율성**: 정규식/문자열 처리로 충분한 성능 확보
