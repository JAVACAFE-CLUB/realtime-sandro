## 검색 시스템 아키텍처 명세 📑

지금까지 논의한 내용을 바탕으로 각 시스템의 역할, 책임, 데이터 흐름을 정의한 상세 명세입니다. 이 문서를 기반으로 각 마이크로서비스 개발을 진행할 수 있습니다.

-----

### \#\# 1. 시스템 개요 (Overall Architecture)

4개의 마이크로서비스(수집, 정제, 색인, 서빙)가 Kafka를 통해 비동기적으로 통신하며, 검색 데이터를 처리 및 제공하는 시스템입니다. 원본 데이터는 영구 보관을 위해 객체 스토리지(MinIO)에 아카이빙합니다.

**데이터 흐름:**
`외부 데이터 소스 → [수집 시스템] → Kafka(raw) → [정제 시스템] → Kafka(refined) → [색인 시스템] → 검색 엔진 → [서빙 시스템] → 사용자`

**영구 저장소:**
`[수집 시스템] → MinIO (원본 데이터 아카이브)`

-----

### \#\# 2. 공통 기술 스택 (Common Tech Stack)

* **언어 및 프레임워크**: Kotlin + Spring Boot
* **메시지 큐**: Apache Kafka
* **객체 스토리지**: MinIO (AWS S3 호환)
* **검색 엔진**: Elasticsearch 또는 OpenSearch

-----

### \#\# 3. Kafka 토픽 명세 (Kafka Topics)

#### \#\#\# 3.1. `raw-documents-topic`

* **목적**: 수집 시스템이 원본에서 추출한 '날것'의 데이터를 정제 시스템으로 전달.
* **메시지 포맷 (JSON Schema)**:
  ```json
  {
    // 메시지 고유 ID (UUID, 수집 시스템에서 생성)
    "messageId": "string", 
    // 데이터 소스 (WIKI, NEWS, X_POST 등)
    "sourceType": "string", 
    // 데이터 수집 시간 (ISO 8601)
    "collectedAt": "string", 
    // MinIO에 저장된 원본 데이터의 경로 (URI)
    "rawDataLocation": "string",
    // 정제 시스템이 처리할 실제 데이터
    "payload": {
      // 소스별로 구조가 다름 (아래 예시)
    }
  }
  ```
    * **`payload` 예시 (`sourceType: "WIKI"`)**
      ```json
      "payload": {
        "title": "string",
        "rawContent": "string" // <page>의 text 내용
      }
      ```

#### \#\#\# 3.2. `refined-documents-topic`

* **목적**: 정제 시스템이 처리를 마친, 메타데이터가 풍부한 문서를 색인 시스템으로 전달.
* **메시지 포맷 (JSON Schema)**:
  ```json
  {
    // 문서 고유 ID (messageId와 동일하게 사용 가능)
    "documentId": "string",
    // 데이터 출처
    "sourceType": "string",
    // 원문으로 연결될 표준 URL
    "url": "string",

    // --- 검색 대상 필드 ---
    "title": "string",
    "contentText": "string", // HTML/마크업이 제거된 순수 텍스트

    // --- 필터링/정렬용 필드 ---
    "createdAt": "string", // 콘텐츠 원본 생성/발행일 (ISO 8601)
    
    // --- 패싯/분석용 메타데이터 ---
    "metadata": {
      // sourceType에 따라 해당 객체에만 데이터가 채워짐
      "wiki": {
        "lastModified": "string",
        "categories": ["string", ...]
      },
      "news": {
        "author": "string",
        "press": "string"
      },
      "x_post": {
        "author": "string",
        "hashtags": ["string", ...]
      }
    }
  }
  ```

### \#\# 4. 시스템별 상세 명세

#### \#\#\# 4.1. 수집 시스템 (Collection Service) 📥

* **역할**: 다양한 외부 소스로부터 데이터를 가져와 Kafka에 전송하고, 원본을 MinIO에 아카이빙한다.
* **입력**: 위키피디아 덤프 파일 (`.xml.bz2`), 뉴스 RSS 피드, X API 등.
* **주요 처리 로직**:
    1. **데이터 소스별 스케줄러/리스너 실행**:
        * (위키) 지정된 경로의 덤프 파일을 읽기 시작.
        * (뉴스) 주기적으로 RSS 피드를 확인.
    2. **스트리밍 처리 (위키 덤프)**:
        * `.bz2` 압축 스트림과 XML 스트리밍 파서를 연결.
        * **Jackson XML 데이터 바인딩**을 사용해 `<page>` 단위로 XML을 코틀린 데이터 클래스에 자동으로 매핑. 메모리 부하 최소화.
    3. **데이터 처리 단위 생성**:
        * 각 `<page>`, 뉴스 기사, X 게시물에 대해 고유 `messageId`(UUID)를 생성.
    4. **원본 아카이빙**:
        * 처리 단위의 완전한 원본(e.g., `<page>` XML 전체)을 MinIO에 저장. 파일명은 `messageId`를 사용.
    5. **Kafka 메시지 생성 및 전송**:
        * `raw-documents-topic` 명세에 맞는 메시지를 생성.
        * `payload`에는 정제에 필요한 최소한의 데이터(`title`, `rawContent`)를 담는다.
        * Kafka Producer를 통해 메시지를 전송.
* **출력**:
    * Kafka `raw-documents-topic`으로의 메시지.
    * MinIO에 저장된 원본 데이터 파일.
* **주요 기술/라이브러리**: `Apache Commons Compress`, `Jackson-dataformat-xml`, `Spring Kafka`.

#### \#\#\# 4.2. 정제 시스템 (Refinement Service) ✨

* **역할**: Kafka 메시지를 받아 원본 콘텐츠를 파싱하고, 의미 있는 텍스트와 메타데이터를 추출하여 구조화된 최종 문서를 만든다.
* **입력**: Kafka `raw-documents-topic`의 메시지.
* **주요 처리 로직**:
    1. **메시지 소비**: Kafka Consumer가 `raw-documents-topic`에서 메시지를 가져온다.
    2. **프로세서 분기**: 메시지의 `sourceType` 필드를 확인하여 데이터 소스에 맞는 처리 로직을 선택한다.
    3. **콘텐츠 파싱 및 정제**:
        * (위키) `payload.rawContent`의 위키텍스트를 파싱하여 순수 텍스트 추출.
        * (뉴스) `payload.rawContent`의 HTML을 파싱하여 본문 텍스트 추출.
    4. **메타데이터 추출**:
        * (위키) 카테고리, 최종 수정일 등 추출.
        * (뉴스) 기자명, 언론사, 발행일 등 추출.
    5. **Kafka 메시지 생성 및 전송**:
        * `refined-documents-topic` 명세에 맞는 풍부한 메시지를 생성.
        * 추출한 모든 메타데이터와 정제된 텍스트를 적절한 필드에 채운다.
        * Kafka Producer를 통해 메시지를 전송.
* **출력**: Kafka `refined-documents-topic`으로의 메시지.
* **주요 기술/라이브러리**: `Jsoup`(HTML 파싱), 미디어위키 파서 라이브러리, `Spring Kafka`.

#### \#\#\# 4.3. 색인 시스템 (Indexing Service) ✍️

* **역할**: 정제된 문서를 Kafka에서 받아 검색 엔진에 저장(색인)한다.
* **입력**: Kafka `refined-documents-topic`의 메시지.
* **주요 처리 로직**:
    1. **메시지 소비**: `refined-documents-topic`에서 정제된 문서 메시지를 가져온다.
    2. **데이터 변환**: 메시지(JSON)를 검색 엔진의 색인 API가 요구하는 포맷으로 변환한다.
    3. **데이터 색인**: 검색 엔진 클라이언트를 통해 문서를 색인 요청. `documentId`를 문서의 `_id`로 사용.
    4. **오류 처리**: 색인 실패 시, 재시도 로직을 수행하거나 해당 메시지를 별도의 Dead Letter Queue(DLQ) 토픽으로 보내 분석할 수 있도록 한다.
* **출력**: 검색 엔진에 저장된 문서 데이터.
* **주요 기술/라이브러리**: Elasticsearch/OpenSearch Client, `Spring Kafka`.

#### \#\#\# 4.4. 서빙 시스템 (Serving Service) 🔍

* **역할**: 사용자로부터 검색 쿼리를 받아 검색 엔진에 질의하고, 그 결과를 API 형태로 제공한다.
* **입력**: 사용자의 HTTP 요청 (e.g., `GET /api/search?q=검색어`).
* **주요 처리 로직**:
    1. **API 요청 접수**: REST Controller가 사용자 쿼리 및 파라미터(페이지 번호, 필터 등)를 받는다.
    2. **검색 쿼리 생성**: 입력된 사용자 쿼리를 검색 엔진이 이해할 수 있는 DSL(Domain Specific Language) 쿼리로 변환.
        * `title` 필드에 더 높은 가중치 부여.
        * 카테고리, 작성일 등으로 필터링하는 조건 추가.
        * 검색어 하이라이팅 설정.
    3. **검색 엔진 질의**: 검색 엔진 클라이언트를 통해 생성된 쿼리를 실행.
    4. **결과 포맷팅**: 검색 엔진으로부터 받은 결과를 API 응답 규격에 맞게 가공.
        * 불필요한 필드 제거.
        * 페이지네이션 정보(총 개수, 현재 페이지 등) 추가.
    5. **API 응답 반환**: 최종적으로 가공된 JSON 데이터를 사용자에게 반환.
* **출력**: JSON 형태의 API 응답.
* **주요 기술/라이브러리**: `Spring WebFlux/MVC`, Elasticsearch/OpenSearch Client.
